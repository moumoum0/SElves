## 2. 启动与事务中使用 `runBlocking` / `runInTransaction + runBlocking`，有卡顿与死锁风险

文件：
- [app/src/main/java/com/selves/xnn/MainActivity.kt](cci:7://file:///d:/project/selves/app/src/main/java/com/selves/xnn/MainActivity.kt:0:0-0:0)

### 2.1 [MainActivity.attachBaseContext()](cci:1://file:///d:/project/selves/app/src/main/java/com/selves/xnn/MainActivity.kt:44:4-57:5) 中使用 `runBlocking`

```kotlin
val languageCode = runBlocking {
    memberPrefs.language.first()
}
```

问题：

- [attachBaseContext()](cci:1://file:///d:/project/selves/app/src/main/java/com/selves/xnn/MainActivity.kt:44:4-57:5) 是启动关键路径
- 在这里阻塞读取 DataStore，容易增加冷启动延迟
- 如果后续语言逻辑变复杂，会让启动更脆弱

结论：

- 这不是一定崩，但属于**高风险启动阻塞点**

### 2.2 [BackupService.importDataToDatabase()](cci:1://file:///d:/project/selves/app/src/main/java/com/selves/xnn/data/BackupService.kt:688:4-787:5) 中：

```kotlin
database.runInTransaction {
    kotlinx.coroutines.runBlocking {
        ...
    }
}
```

问题：

- 在 Room 事务里再套 `runBlocking` 是很危险的设计
- 事务应该尽量同步、短时、无阻塞地完成
- 这种写法非常容易在未来演化中引入：
  - 卡死
  - 隐式线程阻塞
  - 导入大数据时性能急剧下降
  - 更难定位事务失败原因

建议：

- 改成纯同步 DAO 调用事务
- 或者使用 Room 官方推荐的协程事务方式，而不是事务里包 `runBlocking`

## 4. 权限策略明显过重，存在审核与隐私风险

文件：
- [app/src/main/AndroidManifest.xml](cci:7://file:///d:/project/selves/app/src/main/AndroidManifest.xml:0:0-0:0)

高风险权限包括：

- `MANAGE_EXTERNAL_STORAGE`
- `QUERY_PACKAGES`
- `com.android.permission.GET_INSTALLED_APPS`
- `ACCESS_BACKGROUND_LOCATION`
- `SCHEDULE_EXACT_ALARM`
- `USE_EXACT_ALARM`
- `requestLegacyExternalStorage="true"`

问题：

- 你的实际备份导入导出已经走了 `CreateDocument/OpenDocument`
- 这类 SAF 流程通常**不需要全盘存储权限**
- 对地图检测、应用列表、后台定位、精确闹钟的声明都属于**商店审核高敏感项**
- 特别是：
  - `MANAGE_EXTERNAL_STORAGE` 基本是高危声明
  - `GET_INSTALLED_APPS` 极其敏感
  - `requestLegacyExternalStorage` 已经过时
  - `USE_EXACT_ALARM` 不是随便就该声明的

结论：

- 这是我看到的**最明显的合规风险点**

建议：

- 最小权限原则
- 能用 SAF 的地方，不要申请文件管理权限
- 只有功能确实需要且 UI 已完整申请引导时，再保留后台定位/精确闹钟
- `queries` 比 `QUERY_PACKAGES` 更合理，但也应尽量克制

## 4. 权限策略明显过重，存在审核与隐私风险

文件：
- [app/src/main/AndroidManifest.xml](cci:7://file:///d:/project/selves/app/src/main/AndroidManifest.xml:0:0-0:0)

高风险权限包括：

- `MANAGE_EXTERNAL_STORAGE`
- `QUERY_PACKAGES`
- `com.android.permission.GET_INSTALLED_APPS`
- `ACCESS_BACKGROUND_LOCATION`
- `SCHEDULE_EXACT_ALARM`
- `USE_EXACT_ALARM`

问题：

  - `requestLegacyExternalStorage` 已经过时
  - `USE_EXACT_ALARM` 不是随便就该声明的

结论：

- 这是我看到的**最明显的合规风险点**

建议：

- 最小权限原则
- 能用 SAF 的地方，不要申请文件管理权限
- 只有功能确实需要且 UI 已完整申请引导时，再保留后台定位/精确闹钟
